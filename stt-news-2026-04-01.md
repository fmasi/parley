## Details

### 1. Cohere Transcribe — New SOTA Open-Source ASR

Cohere released **Transcribe**, a 2B-parameter encoder-decoder model (Fast-Conformer encoder, X-attention transformer) under Apache 2.0. It achieves a WER of 5.42 on the HuggingFace Open ASR leaderboard, beating Qwen3-ASR-1.7B, ElevenLabs Scribe v2, and IBM Granite 4.0 1B. It supports 14 languages (EN, FR, DE, IT, ES, PT, EL, NL, PL, ZH, JA, KO, VI, AR) and can process 525 minutes of audio per minute. The model is available on HuggingFace and via Cohere's API for free.

**Sources:** [TechCrunch](https://techcrunch.com/2026/03/26/cohere-launches-an-open-source-voice-model-specifically-for-transcription/) · [Cohere Blog](https://cohere.com/blog/transcribe) · [HuggingFace](https://huggingface.co/CohereLabs/cohere-transcribe-03-2026)

**Relevance:** At 2B parameters, Transcribe is too large for comfortable on-device inference on M-series chips. However, it validates that encoder-decoder architectures with careful training can beat much larger models. Its language coverage overlaps well with the Transcriber app's target languages (FR, PT, ES, EN, KO, JA are all covered). Worth monitoring for potential distilled/quantized variants.

### 2. Argmax Pro SDK for Android — Cross-Platform SpeakerKit

Argmax made its Pro SDK generally available on Android (Kotlin-first, built on Google LiteRT) on March 18. WhisperKit, TTSKit, and SpeakerKit are now available cross-platform starting with SDK v0.9.0+. Argmax SDK 2, powered by Nvidia Sortformer, represents a generational accuracy leap for speaker diarization. SpeakerKit Pro processes 4 minutes of audio in ~1 second on iPhone with error rates matching pyannote across 13 datasets.

**Sources:** [Argmax Blog](https://www.argmaxinc.com/blog/pro-sdk-ga) · [SpeakerKit](https://www.argmaxinc.com/blog/speakerkit) · [Argmax SDK 2](https://www.argmaxinc.com/blog/argmax-sdk-2)

**Relevance:** SpeakerKit/SDK 2 is a direct potential replacement for pyannote.audio in the Transcriber app's diarization pipeline. The Swift SDK integrates natively, runs on CoreML/ANE, and is dramatically faster. The Sortformer-based SDK 2 promises better accuracy. The $0.42/device/month pricing model needs evaluation against the current free pyannote setup.

### 3. Qwen3-ASR MLX Port — Whisper Challenger on Apple Silicon

The community-driven `mlx-qwen3-asr` project provides a full MLX reimplementation (not a wrapper) of Qwen3-ASR, running natively on M1–M4 hardware. The 0.6B model achieves RTF 0.06 on M2 Max (40% faster than Whisper large-v3 via whisper.cpp at RTF 0.10) with comparable accuracy across 52 languages. 8-bit quantization adds only +0.04pp WER; 4-bit trades +0.43pp WER for maximum speed.

**Sources:** [GitHub: mlx-qwen3-asr](https://github.com/moona3k/mlx-qwen3-asr) · [PyPI](https://pypi.org/project/mlx-qwen3-asr/) · [Northflank Benchmarks](https://northflank.com/blog/best-open-source-speech-to-text-stt-model-in-2026-benchmarks)

**Relevance:** This is the most promising drop-in replacement for mlx-whisper in the Transcriber app. At 0.6B parameters (vs. Whisper large-v3's 1.55B), it's smaller, faster, and matches or exceeds accuracy. The Python MLX package means it could potentially replace mlx-whisper in `transcribe.py` with minimal architecture changes. The 8-bit quantized variant offers an excellent speed/quality tradeoff.

### 4. FluidAudio — CoreML Speaker Diarization in Swift

FluidInference's FluidAudio SDK provides CoreML-optimized speaker diarization based on pyannote's segmentation model + WeSpeaker embeddings + VBx clustering. The CoreML conversion yields 10× speedup on CPU and 20× on GPU vs. PyTorch. The Swift SDK supports macOS 14.0+ and iOS 17.0+ with MIT/Apache 2.0 licensing. The project has 44 releases and is actively maintained.

**Sources:** [GitHub: FluidAudio](https://github.com/FluidInference/FluidAudio) · [HuggingFace: speaker-diarization-coreml](https://huggingface.co/FluidInference/speaker-diarization-coreml) · [inference.plus](https://inference.plus/p/low-latency-speaker-diarization-on)

**Relevance:** This is directly applicable to the Transcriber app. FluidAudio uses the same pyannote segmentation + WeSpeaker architecture that pyannote.audio uses, but converted to CoreML for native Swift integration. This could eventually replace the Python pyannote dependency entirely, making the app a pure Swift solution. The 10–20× speedup would dramatically reduce post-recording processing time.

### 5. Soniqo speech-swift — All-in-One MLX Audio Toolkit

Soniqo released diarization and VAD support in February 2026, adding Silero VAD v5, pyannote segmentation 3.0, and WeSpeaker ResNet34 to their Swift MLX toolkit. The library also includes Qwen3-ASR for transcription, CosyVoice/Kokoro TTS, and speech-to-speech via PersonaPlex. Everything runs locally on Apple Silicon.

**Sources:** [GitHub: speech-swift](https://github.com/soniqo/speech-swift) · [Soniqo](https://soniqo.audio) · [Medium: Diarization on Apple Silicon](https://blog.ivan.digital/speaker-diarization-and-voice-activity-detection-on-apple-silicon-native-swift-with-mlx-92ea0c9aca0f)

**Relevance:** speech-swift bundles ASR (Qwen3-ASR) + diarization (pyannote/WeSpeaker) + VAD (Silero) in a single Swift package — essentially what the Transcriber app does with separate Python and Swift components. Worth evaluating as a potential path to a fully native Swift transcription pipeline.

### 6. Moonshine "Flavors" — Tiny Monolingual ASR Models

Moonshine released specialized 27M-parameter monolingual models for Arabic, Chinese, Japanese, Korean, Ukrainian, and Vietnamese. These tiny models outperform Whisper Small (9×larger) and match Whisper Medium (28× larger) by training on carefully balanced mixes of human-labeled, pseudo-labeled, and synthetic data. Error rates are 48% lower than Whisper Tiny.

**Sources:** [arXiv: 2509.02523](https://arxiv.org/abs/2509.02523) · [GitHub: moonshine](https://github.com/moonshine-ai/moonshine) · [HuggingFace](https://huggingface.co/UsefulSensors/moonshine)

**Relevance:** The monolingual approach is interesting for the Transcriber app's multilingual needs (FR, PT, ES, EN, KO, JA). Rather than one large multilingual model, a router could select a tiny language-specific model for better accuracy at lower compute. However, Moonshine doesn't yet cover French, Portuguese, Spanish, Turkish, Finnish, or Swedish — key Transcriber languages.

### 7. whisper.cpp — Incremental Improvements

whisper.cpp continues steady development with updates through March 29. Recent changes include Metal backend performance gains, refreshed iOS/macOS examples with a new XCFramework workflow for easier integration, and build/CI improvements.

**Sources:** [GitHub Releases](https://github.com/ggml-org/whisper.cpp/releases) · [GitHub](https://github.com/ggml-org/whisper.cpp)

**Relevance:** The XCFramework improvements make whisper.cpp easier to integrate into native macOS apps. However, Qwen3-ASR via MLX now outperforms whisper.cpp on M-series chips (RTF 0.06 vs 0.10), reducing the incentive to switch from mlx-whisper to whisper.cpp.

## Action Items

1. **Evaluate Qwen3-ASR via mlx-qwen3-asr** — Most promising near-term improvement. Test the 0.6B 8-bit model against mlx-whisper large-v3 on representative meeting audio (mixed EN/FR, multiple speakers). Compare WER, RTF, and memory usage on target M-series hardware. Integration effort should be low since both are Python MLX packages.

2. **Evaluate FluidAudio for native diarization** — Test FluidAudio's CoreML diarization against pyannote.audio on the same meeting recordings. If quality is comparable, this could eliminate the Python dependency for diarization and dramatically speed up processing. Check macOS 14 compatibility requirement vs. the app's macOS 15 minimum.

3. **Monitor Cohere Transcribe for distilled variants** — At 2B parameters it's too heavy for on-device use today, but if Cohere or the community releases distilled/quantized versions (especially for MLX), it could be a strong candidate given its SOTA WER scores.

4. **Track Argmax SDK 2 pricing and capabilities** — SpeakerKit Pro with Sortformer could be a premium diarization option. Compare accuracy against pyannote 3.1 and FluidAudio on real meeting data. The per-device pricing model may or may not be acceptable.

5. **Investigate speech-swift as a long-term architecture target** — A fully native Swift pipeline (Qwen3-ASR + pyannote diarization + Silero VAD, all via MLX) could eliminate the Python dependency entirely. This is a larger architectural change but worth prototyping.

6. **No action needed on Moonshine** — Language coverage doesn't yet match needs. Revisit when French/Portuguese/Spanish models are available.
